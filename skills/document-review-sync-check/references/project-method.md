# 文档审查方法 / Document Review Method

> 面向本项目的文档审查 SOP。目标是系统性检查一份文档是否与其关联文档、实际目录、配置文件和资源资产保持同步，并输出可执行的修复建议。

---

## 1. 为什么需要这份方法文档

本项目的设计文档生态具备以下特征：

- 文档数量多，且存在明显的上下游关系（schema → 枚举表 → 美术需求 → 接入指南 → 实际资产）
- 文档经常发生重命名、目录迁移、结构重构
- 运行时真实状态不只在文档中，还分散在 `json`、目录结构、资源文件、配置文件中
- 单点修文容易造成下游断链或语义漂移

因此，文档审查不能只检查“这份文档自己写得对不对”，而必须检查：

1. 引用是否还存在
2. 参数和字段是否一致
3. 上游变更是否已经传导到下游
4. 实际文件系统/资源资产是否与文档描述匹配

---

## 2. 适用范围

本方法适用于以下文档类型：

- 功能设计文档
- 需求文档
- 枚举表 / 配置表
- 接口契约文档
- 资源规范文档
- 接入指南文档
- 资源枚举 JSON / Markdown 对照表

尤其适合以下场景：

- 文档改名 / 目录重构后做全链路同步检查
- schema 字段变更后检查下游派生文档
- JSON / Markdown / 实际资产三方核对
- 审查“是否与关联文档保持同步”类需求

---

## 3. 三层检查法

本项目推荐使用“三层检查法”进行文档同步审查：

### L1 — 引用存在性

检查引用的对象是否还存在。

#### 检查内容
- Markdown 链接路径是否存在
- 被引用文档是否已改名 / 删除
- 章节锚点是否仍然有效
- 引用的目录 / 文件是否还在

#### 常见问题
- 文档已重命名，但旧链接未更新
- 子目录多了一层，`../` 与 `../../` 写错
- 仍引用已废弃的旧文档

#### 输出标准
- 路径存在：通过
- 路径不存在：R1
- 路径存在但引用对象已废弃：R2

---

### L2 — 参数一致性

检查内容定义是否一致，而不是只看文件是否存在。

#### 检查内容
- 字段名是否一致
- 枚举值是否一致
- 版本号是否一致
- `id / ap_cost / range / aoe / target_filter / skill_effect` 是否一致
- 资源命名规范是否一致
- 文档中的示例是否仍符合当前实现

#### 典型对象
- `#200 skill_system.md` vs `#109 枚举表`
- `skills.json` vs 枚举表
- `Skill-VFX-Art-Requirements.md` vs `Skill-VFX-Integration-Guide.md`
- 资源目录 vs 资源表 JSON

#### 输出标准
- 字段/数值/枚举完全一致：通过
- 轻微版本滞后但不影响含义：R3
- 字段/数值/枚举冲突：R1 / R2（按影响范围）

---

### L3 — 反向归因

检查上游变更是否已经传导到所有下游文档和资产。

#### 核心问题
> 如果 A 文档改了，所有引用/依赖 A 的文档、JSON、目录结构、资源资产，是否都同步了？

#### 检查内容
- schema 变更是否已经同步到枚举表 / JSON / 接入文档
- 目录重构是否已经同步到接入逻辑 / 文档路径 / 资源引用
- 文档改名是否已经同步到所有引用处
- 资产命名规范变化是否已经同步到 JSON / 代码示例 / 指南文档

#### 输出标准
- 变更已全量传导：通过
- 部分传导：R2
- 关键下游完全未同步：R1

---

## 4. 详细操作步骤

### 步骤 1：确定权威源

先回答：

1. 这份文档是在定义规则，还是在列实例？
2. 它的上游是谁？
3. 它哪些字段自己负责，哪些字段只是引用别人的？

#### 常见角色划分
- schema / 字段权威源：定义字段和枚举
- 枚举表：列实例值
- 资源规范：定义资源命名与交付规则
- 接入文档：定义运行时如何消费
- JSON：程序权威数据源

---

### 步骤 2：检查头表

必看：

- 版本
- 来源文档
- 关联文档
- 前置依赖
- 权威范围
- 负责方

#### 检查目标
- 版本号是否滞后
- 来源文档是否已改名 / 删除
- 权威范围声明是否仍然成立
- 依赖版本是否与实际一致

---

### 步骤 3：做 L1 引用存在性检查

#### 检查方式
- 提取全部 Markdown 链接
- 校验相对路径
- 校验目录深度
- 确认锚点对象是否还存在

#### 重点注意
- 文档路径改名
- 目录层级变化
- 旧版 / 已废弃文档残留引用

---

### 步骤 4：做 L2 参数一致性检查

#### 典型对照项
- 版本号
- 字段定义表
- 枚举值集合
- 数据值（AP / 射程 / AOE / target_filter / skill_effect）
- 资源命名
- 资源计数
- 示例 JSON / 示例代码

#### 核心原则
- 逐字段核对
- 逐行核对
- 示例也算正文，不能跳过

---

### 步骤 5：做 L3 反向归因检查

#### 做法
从上游改动出发，逐个找下游：

- 谁引用了它
- 谁依赖了它
- 谁使用了它的字段/命名/目录规则
- 谁把它当作权威源

然后逐个回查是否已同步。

---

### 步骤 6：对照实际文件系统 / 资源资产

文档检查不能停留在文档内，必须对照实际仓库：

#### 需要核对
- 目录是否存在
- 文件是否存在
- JSON 是否可解析
- 资源数量是否匹配
- 资源命名是否匹配
- 文档写“已交付”的东西是否真的已交付

#### 特别提醒
要区分以下不同层次：
- `normalized/`：可运行的资源资产
- `source/`：原始素材 / UI 图标 / 中间文件
- `icon/`：技能 icon
- `postfx.json`：后处理资源
- `vfx-assets.json`：资源枚举表

---

### 步骤 7：区分“内容问题”和“落地问题”

#### 内容问题
文档之间互相矛盾，例如：
- 单体 vs AOE
- T3 vs T4
- 旧字段 vs 新字段

#### 落地问题
文档是对的，但现实还没跟上，例如：
- 资源未交付
- JSON 未同步
- 目录已改但指南还没改

#### 处理原则
- 内容问题：必须修文档
- 落地问题：明确写“当前状态 / 待同步 / 待交付”，不能伪装成已完成

---

### 步骤 8：按近源原则修复

优先修距离权威源最近的地方：

1. 先修 schema / 上游定义
2. 再修主枚举表
3. 再修接入文档 / 资源文档
4. 最后修示例、附录、引用说明

避免只在下游补注释而不修上游定义。

---

### 步骤 9：回归验证

至少完成以下复核：

- 链接全部可达
- 版本号全部正确
- 字段定义一致
- 关键数据一致
- 目录 / 文件存在
- JSON 可解析
- 权威范围没有写假话

---

## 5. 本项目高频问题清单

### 5.1 文档改名未传导

典型模式：
- `skill-system-frs.md` 已改名成 `skill-effect-design.md`
- 但多个下游文档仍引用旧文件名

### 5.2 目录层级变化导致断链

典型模式：
- 文档从 `skill-system/` 下挪到 `skill-effects/` 子目录后，`../buff-system/...` 应改为 `../../buff-system/...`

### 5.3 schema 已变更，下游仍使用旧字段

典型模式：
- `#200` 从扁平 VFX 字段切到 `vfx_config`
- 下游仍写 `cast_vfx_id` / `hit_vfx_id`

### 5.4 资源文档与接入文档脱节

典型模式：
- 资源表说“查表驱动”
- 接入文档仍然“直接拼路径”

### 5.5 JSON / Markdown / 实际资产三方不同步

典型模式：
- 文档说有 32 项
- JSON 只有 31 项
- 磁盘上又是另一套目录结构

### 5.6 icon / source / normalized 混淆

- `icon/` 不应混入 VFX 主资源表
- `source/` 不等于运行时资产
- `normalized/` 才是当前运行时可消费资源

---

## 6. 修复优先级原则

| 优先级 | 处理对象 | 原则 |
|------|---------|------|
| P0 | 权威源错误 / 断链 / 关键字段冲突 | 先修 |
| P1 | 下游派生文档未同步 | 紧随其后 |
| P2 | 示例、附录、版本说明 | 在主链路正确后修 |
| P3 | 风格与措辞优化 | 最后处理 |

---

## 7. 验证清单

审查或修复完成后，逐项核对：

- [ ] 权威源已确认
- [ ] 头表版本 / 来源 / 关联 / 依赖无误
- [ ] 全部链接可达
- [ ] 关键字段定义一致
- [ ] 核心数据与 JSON 一致
- [ ] 实际目录 / 文件 / 资源存在
- [ ] 示例未过期
- [ ] 资源交付状态与文档描述一致
- [ ] 已区分“内容问题”和“落地问题”
- [ ] 修复采用近源原则
- [ ] 回归验证通过

---

## 8. 输出模板建议

建议审查输出遵循以下结构：

1. **结论**：是否同步（完全同步 / 基本同步 / 不同步）
2. **已同步项**：列清楚哪些维度无问题
3. **问题清单**：逐条列出位置、问题、影响范围
4. **问题分类**：内容问题 / 落地问题
5. **修复建议**：先修什么、后修什么
6. **验证结果**：修复后再回归确认

如需正式报告，可复用：
- `templates/review-report.md`

---

## 9. 一句话方法总结

> 先确定权威源，再做 L1 引用存在性、L2 参数一致性、L3 反向归因三层检查，最后对照真实文件系统与 JSON 回归验证。
