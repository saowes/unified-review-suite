# Review Orchestrator(统一审查编排器)

本技能是"统一审查套件 unified-review-suite"的唯一入口。它本身不重复实现具体审查逻辑，而是
**判断目标 → 路由到对应能力 → 串成流水线 → 汇总统一报告**。

## 第一步：识别审查目标

根据用户意图与变更内容判断本次属于哪条线，并据此选择流水线：

| 目标类型 | 触发信号 | 走哪条线 |
| --- | --- | --- |
| 文档一致性 | 涉及 .md / schema / 枚举表 / 接入文档 / JSON / 资源资产；询问文档是否同步、断链、字段漂移 | A 文档线 |
| 代码质量与安全 | 涉及源码改动、PR、上线前检查、安全/漏洞/bug | B 代码线 |
| 全量 | 用户说"完整/统一/全套审查"，或改动同时含文档与代码 | A + B |

无法确定时，先问一句目标范围，再继续，不要盲目跑全量。

## 第二步：执行流水线

### A 文档线 —— 套用 document-review-sync-check

读取并应用同套件中的 `document-review-sync-check` 技能，按其三层方法执行：
- **L1 引用存在性**：链接、被引用文件/目录是否存在；改名删除是否仍被引用。
- **L2 参数一致性**：字段名、枚举值、版本号、JSON 结构、示例片段是否一致。
- **L3 反向传导**：上游改动是否已传导到所有下游文档。
- 必须把结论区分为"内容问题(文档互相矛盾)"与"rollout 问题(文档对但实现/资源未跟上)"。

### B 代码线 —— 调度内置审查子代理

通过 Task 工具派起 Cursor 内置子代理(不要在本技能内重写其逻辑)：
- `bugbot`：逻辑缺陷、bug、边界条件。
- `security-review`：安全漏洞、注入、鉴权、敏感信息。
- `code-reviewer`：质量、可维护性、规范一致性。

按需选择(默认全量审查时三者都跑)。

**回退策略**：若当前环境不提供这些内置子代理，则降级为由本技能直接对照通用清单进行
代码审查(正确性/边界/安全/可读性/测试覆盖)，并在报告中标注"已降级：未使用内置子代理"。

### 第三步：验证收尾(借鉴 verification-before-completion)

任何"已修复/已通过"的结论必须有证据：重跑相关检查(链接、版本、字段、JSON 解析、真实文件
存在性、测试)，确认输出后再下结论。禁止仅凭推断声称完成。

## 统一输出报告模板

无论走哪条线，最终汇总为同一结构(分级口径与 `unified-review` 规则一致)：

```text
# 统一审查报告

## 结论
完全同步/通过 · 大体通过(有非阻断项) · 不通过

## 已通过的维度
- ...

## 发现的问题(按严重度)
- [P0] 阻断/权威源/安全高危 — 位置 · 问题 · 影响 · 内容问题还是 rollout 问题
- [P1] 下游文档未更新 / 重要 bug — ...
- [P2] 示例/附录/版本说明陈旧 — ...
- [P3] 风格类 — ...

## 修复优先级与建议(就近源头修复)
1. 权威源 → 2. 主要派生文档 → 3. 接入指南 → 4. 示例/附录

## 验证结果
- 复检了哪些项、结论
```

## 严重度分级(P0–P3)

- **P0**：阻断性。断链到权威源、安全高危、文档与运行时严重矛盾。
- **P1**：重要。下游文档未随上游更新、明确 bug、字段漂移。
- **P2**：次要。示例/附录/版本号陈旧。
- **P3**：风格。措辞、格式。

## 边界与原则

- 编排器**依赖**同套件的 `document-review-sync-check` 与内置代码审查子代理；缺失时按回退策略降级并标注。
- 不抢占子技能的独立触发：单独的文档审查或代码审查仍可被直接调用。
- 不给绝对化结论；区分"未交付"与"文档写错"。

## 相关组件(同套件)

- 文档审查方法详见 `document-review-sync-check/SKILL.md`。
- 共享分级与模板由规则 `unified-review.mdc` 固化。
- 手动一键发起见命令 `/review`。
